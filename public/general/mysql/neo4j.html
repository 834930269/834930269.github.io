# 标签

我们常用一个标签来代替一组数据集,标签随着创建节点时添加的标签信息一同创建.

例如,你们可以指定一个新的用户通过给节点一个新的标签 :User.当然,你们也可以寻找自己的想要的用户节点,通过指定一个用户名.

标签可以在运行时被移除或者添加,这些节点可以添加一个临时状态,:Suspended 标签可以被用于标记这些.

一个节点可以有一个或多个标签.这群节点被标记为 Person 和 Movie, 但其实我们也可以给一个节点标记更多地的标签.

下面这张图展示了一个节点内可以存在多个标签.

# 关系

一个关系用于连接两个节点,关系的出现使得大量节点可以通过关系变成一个结构. 故,我们允许一个图可以重塑成一个列表,一个树,图,或者一个复合实体(多个节点组合成一个实体)—或者其他的更加复杂的的结构.
下面将展示关系:

# 关系类型

一个关系必须至少包含一个关系.

我们的例子使用ACTED_IN和DIRECTED 作为关系属性,并且ACTED_IN关系类型中包含一个命名为roles的数组属性.

且该关系必须是有向图.

我们可以看到Tom Hanks有一个出边,当Forrest Gump节点则有一个入边.

关系一定存在一个方向,所以,你需要注意哪些方向是有用的,这就意味着尽量不要添加一些没意义或者重复的关系,除非这个关系是必要的.

比如: Tom Hanks Knows himself

# 属性

属性是键值对,可以添加在节点或者关系上.

在我们的示例图中,我们将属性name和born添加到Person节点上,title和released添加到Movie节点上,并且将roles属性添加到关系:ACTED_IN上.

这些属性的值可以有不同的类型,如number,string,以及boolean.对于其他的可用类型,参考Cypher语法手册(官网里).

# 遍历和路径

(Traversals and paths)

一个遍历代表的是如何查询一个图来寻找您想要的答案,一般可以用一个通俗的语言来表达,如: "什么音乐我的朋友喜欢,但我还没有拥有." "What music do my friends like that I don't yet own?",或者 "如果电源中断,哪些网站服务会受到影响?" "What web services are affected if this power supply goes down?"

遍历意味着以某种方式(规则)访问节点,通常访问的将是图的一个子集.

如果我们想要找到哪个电影Tom Hanks表演过的,遍历将从Tom Hanks节点开始,紧接着将通过:ACTED_IN关系连接到下一个节点Movie,最终将返回一个Forrest Gump电影 （遍历结束).

# 模式

模式(Schema),在关系数据库的理论中一般是位于数据库之上的一层,通常在关系数据库中模式会分为三层.

外模式: 面向应用程序提供的视图或者存储过程(即一部分数据库的结构)

模式: 整个数据表的集合

内模式: 数据表的实际物理存储和底层数据结构,数据库管理者不需要看到这部分.

在neo4j中,你没有必要去添加模式,但是它是存在的,一个neo4j中的模式依赖于索引(indexes)和限制(constraints).
