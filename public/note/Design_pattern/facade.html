# 外观模式

项目一直在使用外观模式,这个模式给我的感觉是类似于middleware

基本是一个一个小组件做好以后插进去的

至少在Unity中感觉很好用,各个功能分开来做,然后通过facade暴露给主线程

举个栗子,你可以将资源下载/信息控制/Socket/模型加载什么的全部都分开.

这也就导致了一个项目就算再大也很有逻辑,不会说独立开发者超过万行代码就搞不懂逻辑了

# 事多,不想敲代码,直接Ctrl+C/V

# 什么是外观模式

       有些人可能炒过股票，但其实大部分人都不太懂，这种没有足够了解证券知识的情况下做股票是很容易亏钱的，刚开始炒股肯定都会想，如果有个懂行的帮帮手就好，其实基金就是个好帮手，支付宝里就有许多的基金，它将投资者分散的资金集中起来，交由专业的经理人进行管理，投资于股票、债券、外汇等领域，而基金投资的收益归持有者所有，管理机构收取一定比例的托管管理费用。

       其实本篇要说的这个设计模式就和这很有关系，由于当投资者自己买股票时，由于众多投资者对众多股票的联系太多，反而不利于操作，这在软件中就成为耦合性太高，而有了基金后，就变成众多用户只和基金打交道，关心基金的上涨和下跌，而实际上的操作确是基金经理人与股票和其它投资产品打交道，这就是外观模式。

# 大体逻辑

如上图,facade作为主系统直接访问的对象,其他的功能则拆分成不同的Subsystem,每个Subsystem负责一个单独的模块.
之后facde通过向外部提供subsystem的接口,就可以直接使外部操作者忽略subsystem的内部结构,直接使用ss的功能.

# 以代码为例

public class Facade {

    //被委托的对象
    SubSystemA a;
    SubSystemB b;
    SubSystemC c;
    SubSystemD d;

    public Facade() {
        a = new SubSystemA();
        b = new SubSystemB();
        c = new SubSystemC();
        d = new SubSystemD();
    }

    //提供给外部访问的方法
    public void methodA() {
        this.a.dosomethingA();
    }

    public void methodB() {
        this.b.dosomethingB();
    }

    public void methodC() {
        this.c.dosomethingC();
    }

    public void methodD() {
        this.d.dosomethingD();
    }

}

如上,但是有点不同的是,一般facade 这个类是采用·单例模式·的,在应用启动时就实例完成,后面的接口暴露逻辑同上面差不多,只需要在facade中注册每个子系统就可以了.
之后facade会通过一个函数调用对应模块的接口.

只不过, 我所接触到的GameFacade不同于该外观模式的一点是,当各个模块实例化的时候,该模块的构造函数会接受到该facade的单例对象.
即任意模块都可以通过facade间接的调用到其他模块的方法,这样灵活性就更强了.
